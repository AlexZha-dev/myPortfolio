# MyPortfolio

Техническое задание на персональный сайт-портфолио для публикации на `GitHub Pages`.

## Назначение

Сайт нужен как публичная витрина проектов, навыков и подхода к разработке.

Основные цели:

- показать лучшие проекты в сильной и понятной подаче;
- кратко рассказать обо мне как о разработчике;
- выделить стек, сильные стороны и рабочий подход;
- дать быстрый доступ к `GitHub`, контактам и демо;
- поддерживать мультиязычность;
- позволять добавлять новый контент без правки JSX-компонентов;
- сохранить простую публикацию как статического сайта.

## Статус локалей

В проекте используются три типа локалей:

- `default locale`: язык-источник, на котором контент обязан существовать всегда;
- `active locales`: языки, которые уже доступны в интерфейсе;
- `planned locales`: языки, архитектурно поддержанные, но ещё не включённые в production UI.

Для текущего этапа:

- `default locale`: `ru`;
- `active locales` для `v1`: `ru`, `en`;
- `planned locale` для следующего этапа: `fi`.

Следствие:

- в `v1` интерфейс обязан полноценно работать на `ru` и `en`;
- структура проекта уже сейчас должна позволять без переделки архитектуры подключить `fi`;
- финский не блокирует релиз `v1`, но должен быть предусмотрен в модели данных, конфигурации локалей и пайплайне контента.

## Канонические решения

Ниже зафиксированы решения, которые считаются основными для проекта.

- приложение находится в папке `portfolio/`;
- стек: `React` + `TypeScript` + `Vite`;
- сайт публикуется как статическая сборка на `GitHub Pages`;
- каноническая папка сборки: `portfolio/dist`;
- деплой выполняется через `.github/workflows/deploy.yml`;
- каноническая release-ветка: `main`;
- интерфейс сайта в версии `v1` является одностраничным;
- навигация внутри страницы выполняется через секции, якоря и deep-link для проектов;
- новый контент проектов и длинных текстовых секций добавляется через `Markdown`;
- `Hero`, `Stack` и `Contacts` допускается хранить в отдельном `typed-config`, так как эти данные меняются редко;
- UI-тексты интерфейса хранятся отдельно от контентных `Markdown`-файлов.

## Область проекта

В первую версию входят:

- главная страница-портфолио;
- переключение языка `RU/EN`;
- архитектурная готовность к подключению `FI`;
- секция проектов с данными из `Markdown`;
- секции `Hero`, `About`, `Stack`, `Projects`, `Process`, `Growth`, `Contacts`;
- адаптивная вёрстка;
- deep-link для открытия конкретного проекта;
- деплой на `GitHub Pages`.

Во второй этап могут войти:

- включение `FI` в production UI;
- расширение числа проектов и переводов;
- выделенные страницы проектов, если это станет необходимо.

В первую версию не входят:

- CMS;
- серверная часть;
- авторизация;
- блог;
- админ-панель;
- комментарии;
- сложный поиск по проектам.

## Архитектурный подход

Проект должен быть реализован как статический frontend-сайт.

Чтобы не усложнять публикацию на `GitHub Pages`, в `v1` сайт проектируется как одностраничное приложение без обязательных вложенных клиентских маршрутов. Это снимает типовые проблемы со статическим хостингом, прямыми переходами по вложенным URL и дополнительной настройкой `404`-fallback.

Полное описание проекта в `v1` открывается внутри той же страницы.

Требования:

- детальный просмотр проекта открывается в модальном окне, drawer или аналогичном слое;
- состояние открытия должно поддерживать deep-link;
- deep-link должен быть реализован через hash или query-параметр без обязательного полноценного роутера;
- при открытии по deep-link пользователь сразу попадает на нужный проект;
- при закрытии карточки URL должен возвращаться в исходное состояние.

Рекомендуемый формат deep-link:

- `#project/<project-id>`

Если в будущем понадобятся отдельные страницы проектов, это должно оформляться как отдельная задача и сопровождаться выбором стратегии маршрутизации для `GitHub Pages`.

## Структура сайта

Сайт должен состоять из следующих секций:

1. `Hero`
2. `About`
3. `Stack`
4. `Projects`
5. `Process`
6. `Growth`
7. `Contacts`

Требования к секциям:

- каждая секция должна иметь понятный заголовок;
- каждая секция должна иметь стабильный `id` для якорной навигации;
- текстовый контент секций должен приходить из контентного слоя или `typed-config`;
- структура должна быть одинаково понятной на всех активных языках;
- при включении `fi` структура должна оставаться такой же без изменения компонентного дерева.

## Источники данных

В проекте используется три источника данных.

### 1. Markdown-контент

Используется для:

- проектов;
- длинных текстов секций `About`, `Process`, `Growth`;
- кейсов и расширенных описаний проектов.

### 2. Typed-config

Используется для:

- `Hero`;
- `Stack`;
- `Contacts`;
- редко меняющихся структурированных данных, которые неудобно хранить в `Markdown`.

### 3. UI-словари переводов

Используются для:

- названий кнопок;
- пунктов навигации;
- коротких системных строк;
- `aria-label`;
- служебных уведомлений;
- бейджей вроде `Translation in progress`.

## Визуальные требования

Сайт должен быть выполнен в тёмно-зелёной стилистике.

Общие требования:

- тёмный фон;
- глубокие зелёные оттенки;
- контрастный, хорошо читаемый текст;
- мягкие акцентные свечения;
- современная и аккуратная визуальная подача;
- адаптивность для мобильных и десктопных экранов.

Базовая палитра:

- фон: `#07110d`
- основной блок: `#0d1b16`
- вторичный блок: `#13241d`
- акцент: `#3fae6f`
- яркий акцент: `#7dffb1`
- текст: `#d8f5e3`

## Мультиязычность

Базовые правила:

- язык по умолчанию: `ru`;
- активные языки `v1`: `ru`, `en`;
- планируемый язык второго этапа: `fi`;
- пользователь может переключить язык через UI;
- выбранный язык должен сохраняться локально;
- переводятся все пользовательские тексты интерфейса;
- контент секций и проектов хранится отдельно по локалям.

Правила полноты перевода:

- для `active locales` словари UI являются обязательными;
- для `active locales` обязательны переводы базовых секций сайта;
- для `planned locales` отсутствие файлов допустимо до момента включения локали;
- отсутствие `fi` не должно ломать `v1`, пока `fi` не переведён в статус `active`.

Политика показа при отсутствии перевода:

- если отсутствует перевод базовой секции для активной локали, сборка должна падать;
- если отсутствует UI-словарь для активной локали, сборка должна падать;
- если отсутствует перевод проекта для активной локали, проект не исключается целиком из сайта;
- в этом случае используется fallback на `default locale`, а в интерфейсе показывается бейдж `Перевод в разработке` или его локализованный аналог;
- для planned locale отсутствие перевода не считается ошибкой сборки.

Это означает:

- проект не исчезает целиком из production;
- пользователь видит контент и понимает, что локальный перевод ещё не завершён;
- сайт не публикует пустые карточки или сломанные разделы.

## Контентная модель

Новый контент должен добавляться без правки JSX-компонентов.

Разделение ответственности:

- проекты и длинные тексты секций хранятся в `Markdown`;
- `Hero`, `Stack` и `Contacts` хранятся в `typed-config`;
- короткие UI-тексты хранятся в словарях переводов;
- компоненты только отображают уже подготовленные данные.

## Структура файлов

Рекомендуемая структура:

```text
portfolio/
  src/
    content/
      projects/
        <project-id>/
          ru.md
          en.md
          fi.md
          cover.webp
          gallery/
            01.webp
            02.webp
      pages/
        about/
          ru.md
          en.md
          fi.md
        process/
          ru.md
          en.md
          fi.md
        growth/
          ru.md
          en.md
          fi.md
    config/
      locale.ts
      hero.ts
      stack.ts
      contacts.ts
    i18n/
      ru.ts
      en.ts
      fi.ts
```

Правила структуры:

- `<project-id>` это стабильный идентификатор проекта;
- файлы `ru.md`, `en.md` и `fi.md` описывают один и тот же проект;
- активные локали перечисляются в `config/locale.ts`;
- `fi.ts` и `fi.md` могут отсутствовать до момента включения `fi` в production;
- изображения проекта лежат рядом с его контентом;
- UI-переводы не должны храниться внутри `Markdown`;
- `typed-config` не должен дублировать содержимое длинных `Markdown`-текстов.

## Locale Config

В проекте должен существовать единый реестр локалей.

Минимальные поля:

- `code`
- `label`
- `status`
- `isDefault`
- `isEnabled`

Допустимые значения `status`:

- `active`
- `planned`

Пример логики:

- `ru`: `active`, `isDefault: true`, `isEnabled: true`
- `en`: `active`, `isDefault: false`, `isEnabled: true`
- `fi`: `planned`, `isDefault: false`, `isEnabled: false`

Все проверки локалей должны опираться именно на этот реестр, а не на захардкоженные списки по коду приложения.

## Typed-config для Hero, Stack и Contacts

### Hero Config

`Hero` хранится в типизированной конфигурации.

Должен содержать:

- имя или ник;
- роль;
- краткое описание по локалям;
- основные CTA;
- ссылки на `GitHub` и другие ключевые точки входа.

### Stack Config

`Stack` хранится в типизированной конфигурации.

Должен состоять из групп, например:

- `frontend`
- `tooling`
- `design`
- `other`

Каждая группа должна содержать:

- `id`
- локализованный `title`
- список элементов

Каждый элемент должен содержать:

- `name`
- опционально `level`
- опционально `note`

### Contacts Config

`Contacts` хранится в типизированной конфигурации.

Каждый контакт должен содержать:

- `id`
- `label`
- `href`
- `type`
- опционально `icon`

Допустимые типы:

- `github`
- `telegram`
- `email`
- `resume`
- `other`

## Формат проекта в Markdown

Каждый проект должен описываться отдельным `Markdown`-файлом для каждой локали.

Обязательные поля frontmatter:

- `title`
- `projectId`
- `locale`
- `summary`
- `stack`
- `featured`
- `sortOrder`
- `status`
- `translationState`
- `cover`
- `coverAlt`

Рекомендуемые поля:

- `repoUrl`
- `demoUrl`
- `tags`
- `publishedAt`
- `year`
- `seoDescription`
- `gallery`

Допустимые значения `status`:

- `draft`
- `published`
- `archived`

Допустимые значения `translationState`:

- `ready`
- `in_progress`

Пример:

```md
---
title: "Portfolio Site"
projectId: "portfolio-site"
locale: "en"
summary: "Personal portfolio website with multilingual content and Markdown-based project management."
stack:
  - React
  - TypeScript
  - Vite
featured: true
sortOrder: 10
status: "published"
translationState: "ready"
repoUrl: "https://github.com/username/myPortfolio"
demoUrl: "https://username.github.io/myPortfolio/"
cover: "./cover.webp"
coverAlt: "Preview of the portfolio website home screen"
tags:
  - portfolio
  - frontend
gallery:
  - src: "./gallery/01.webp"
    alt: "Projects section of the portfolio website"
    caption: "Projects overview"
year: 2026
seoDescription: "Portfolio site built with React and TypeScript."
---

Detailed project description in Markdown.
```

Правила для project frontmatter:

- `projectId` должен совпадать для всех локалей одного проекта;
- `locale` должен быть одним из зарегистрированных кодов локали;
- `sortOrder` используется для ручной сортировки;
- `featured: true` помечает проект как приоритетный;
- `status: "published"` разрешает показ проекта на сайте;
- `draft` не должен отображаться в production;
- `archived` по умолчанию не отображается в основном списке;
- `translationState: "in_progress"` допускается только для не-default локалей;
- `coverAlt` обязателен;
- если указана `gallery`, каждый элемент обязан иметь `src` и `alt`;
- если схема нарушена, сборка должна падать с ошибкой.

## Правила сортировки и отображения проектов

Секция `Projects` должна показывать карточки проектов, собранные из контентного слоя.

В production отображаются только проекты со `status: "published"` или проекты, которые разрешены к показу отдельным правилом.

Порядок сортировки:

1. `featured: true` выше `featured: false`
2. `sortOrder` по возрастанию
3. `publishedAt` по убыванию, если поле задано
4. `title` по алфавиту как последний fallback

Каждая карточка должна уметь отображать:

- название;
- краткое описание;
- стек;
- статус перевода;
- ссылки на `GitHub` и `Live Demo`, если они заданы;
- обложку;
- полный текст проекта по действию пользователя.

Если перевод проекта для активной локали отсутствует:

- используется контент из `default locale`;
- на карточке и в полном описании показывается бейдж `translation in progress`;
- проект остаётся доступным для просмотра;
- карточка не должна выглядеть как ошибка или пустой элемент.

## Формат текстовых секций в Markdown

Для длинных текстовых секций также используются `Markdown`-файлы.

Этот формат применяется минимум к:

- `about`
- `process`
- `growth`

Обязательные поля frontmatter:

- `section`
- `locale`
- `title`
- `translationState`

Рекомендуемые поля:

- `lead`
- `seoDescription`
- `order`

Пример:

```md
---
section: "about"
locale: "ru"
title: "Обо мне"
lead: "Frontend-разработчик, который любит сильный UI и понятную структуру."
translationState: "ready"
---

Основной текст секции в Markdown.
```

Правила для секций:

- для активных локалей секции обязаны существовать;
- `translationState: "in_progress"` для базовых секций активной локали недопустим;
- если секция активной локали отсутствует или невалидна, сборка должна падать.

## Техническая обработка контента

Контентный слой должен быть реализован так, чтобы структура была предсказуемой и валидируемой.

Требования:

- загрузка `Markdown` через `import.meta.glob`;
- разбор frontmatter через `gray-matter` или аналог;
- рендер `Markdown` через `remark`, `react-markdown` или аналог;
- валидация схемы frontmatter через `zod` или аналог;
- единая нормализация данных до слоя UI;
- при ошибке схемы сборка должна завершаться неуспешно.

Что должно валидироваться на сборке:

- корректность frontmatter для проектов;
- корректность frontmatter для секций;
- наличие обязательных `alt`-текстов;
- наличие UI-словарей для всех активных локалей;
- наличие обязательных секций для всех активных локалей;
- корректность locale registry;
- отсутствие дубликатов `projectId`;
- наличие `default locale`-версии у каждого опубликованного проекта.

Это решение нужно для предотвращения следующих проблем:

- разного формата файлов;
- отсутствующих обязательных полей;
- сломанных карточек проектов;
- случайной публикации неполного контента;
- рассинхронизации локалей;
- публикации без доступных alt-текстов.

## Правила добавления нового проекта

Чтобы добавить новый проект, должно быть достаточно:

1. Создать папку `portfolio/src/content/projects/<project-id>/`.
2. Добавить `ru.md`.
3. Добавить `en.md`, если английская локаль уже готова.
4. Добавить `fi.md`, когда финская локаль будет включена или если перевод уже подготовлен заранее.
5. Положить обложку и дополнительные изображения.
6. Заполнить обязательные поля frontmatter, включая `coverAlt`.
7. Указать `status: "published"` после проверки контента.

После этого проект должен появляться на сайте автоматически без правки компонентов.

## SEO и доступность

Минимальные требования:

- корректный `title` страницы;
- `meta description`;
- корректный `lang` на документе;
- alt-тексты для изображений;
- клавиатурная доступность основных интерактивных элементов;
- видимый `focus state`;
- достаточный контраст текста и фона;
- логичная иерархия заголовков;
- локализованные подписи для кнопок и интерактивных элементов.

## Публикация

Сайт публикуется через `GitHub Pages` и `GitHub Actions`.

Зафиксированные правила:

- workflow находится в `.github/workflows/deploy.yml`;
- сборка выполняется из папки `portfolio/`;
- результат сборки публикуется из `portfolio/dist`;
- канонический автоматический деплой срабатывает при `push` в `main`;
- временная поддержка `master` возможна только как переходный режим и не должна считаться целевой конфигурацией.

Если репозиторий публикуется как project site `https://<username>.github.io/<repo>/`, то в `Vite` должен быть корректно задан `base`.

Для текущего репозитория канонический вариант:

- `base: '/myPortfolio/'`

Если тип публикации изменится на user site `<username>.github.io`, значение `base` должно быть пересмотрено отдельно.

## Критерии готовности

Проект считается реализованным корректно, если:

- сайт собирается командой `npm run build`;
- результат сборки находится в `portfolio/dist`;
- workflow публикует сайт на `GitHub Pages`;
- сайт полноценно работает на `ru` и `en`;
- архитектура готова к подключению `fi` без смены модели данных;
- новый проект можно добавить только через `Markdown` и ассеты;
- `Hero`, `Stack` и `Contacts` редактируются через `typed-config`;
- без изменения JSX новый проект появляется в интерфейсе;
- без изменения JSX можно обновить тексты длинных секций;
- при отсутствии перевода проекта для активной локали показывается fallback и бейдж `translation in progress`;
- при отсутствии базовой секции или UI-словаря для активной локали сборка падает;
